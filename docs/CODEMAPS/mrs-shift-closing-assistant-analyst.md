# Помощник закрытия смены: описание для аналитиков

Документ описывает, как устроен помощник закрытия смены в расширении `MRS_ЗакрытиеСмены`: какие объекты участвуют, какие данные читаются, какие проверки выполняются и где смотреть реализацию.

## Назначение

`MRS_ПомощникЗакрытияСмены` — управляемая обработка-мастер для контроля дня по точкам продаж перед закрытием смены.

Основная идея: пользователь выбирает день и точки продаж, затем проходит проверки в понятном порядке:

1. сопоставление объектов фронта;
2. сверка чеков фронта и чеков 1С;
3. формирование и проверка выпусков блюд;
4. отрицательные остатки;
5. итоговый отчет.

Помощник не хранит отдельный бизнес-документ закрытия. Он работает как оркестратор поверх существующих документов, регистров и обработок общепита.

По текущему коду помощник подготавливает день к закрытию и показывает готовность проверок. Не подтверждено, что он сам выполняет финальное закрытие периода или переносит дату запрета изменения.

## Экранная модель

Форма состоит из страницы параметров и группы шагов.

На странице параметров пользователь выбирает:

- `ВыбранныйДень`;
- точки продаж в таблице `СкладыВыбор`.

После команды **Начать работу** выбранные точки продаж сохраняются в табличный реквизит формы `МассивСкладовСессии`. В нем хранятся склад, касса ККМ, фронт-система, код точки продаж и наименование точки продаж.

Дальше форма переключает страницы:

- `СтраницаСопоставления`;
- `СтраницаСверка`;
- `СтраницаФормирование`;
- `СтраницаОстатки`;
- `СтраницаИтоговыйОтчет`.

Статусы шагов пересчитываются фоновыми операциями БСП `ДлительныеОперации` / `ДлительныеОперацииКлиент`.

## Выбор точек продаж

Источник списка точек продаж — запрос формы `ЗаполнитьСкладыВыборНаСервере()`.

Он читает:

- `Справочник.КассыККМ`;
- `ПланОбмена.питУдаленныеКассы`;
- `РегистрСведений.ПЛ_СоответствиеСФронтСистемами`.

Касса попадает в список, если:

- не помечена на удаление;
- связана с кассовым узлом через `КассыККМ.питКассовыйУзел`;
- у узла заполнен `СкладПриготовления`;
- есть запись соответствия, где `ТипОбъекта = КассаККМ`, фронт-система TNG или Micros, код не пустой, а `Объект` указывает на эту кассу.

Это значит, что помощник изначально работает только по уже настроенным точкам продаж.

## Шаг 1. Сопоставление объектов фронта

Основные процедуры:

- `MRS_ПомощникЗакрытияСменыСервер.СтатусШагаСопоставление(День, МассивСкладов)`;
- `MRS_ПомощникЗакрытияСменыСервер.СопоставитьОбъектФронта(ТипОбъекта, ФронтСистема, Код, Объект, Количество)`.

Проверка строит временную таблицу чеков дня из `Документ.питДанныеПродажФронта` с отбором по дню, складам и `ПометкаУдаления = Ложь`.

Затем помощник ищет три категории проблем:

- код точки продаж из шапки чека не сопоставлен с `Справочник.КассыККМ`;
- код кнопки из табличной части `питДанныеПродажФронта.Товары.ПЛ_КодКнопки` не сопоставлен с `Справочник.Номенклатура`;
- в строке чека номенклатура пустая, что считается блокирующей проблемой даже при наличии записи соответствия по коду кнопки.

Сопоставления пишутся в независимый непериодический регистр сведений `ПЛ_СоответствиеСФронтСистемами`.

Ключевые поля регистра:

| Поле | Тип/смысл |
|---|---|
| `ТипОбъекта` | тип объекта фронта: касса ККМ, номенклатура и т.п. |
| `ФронтСистема` | TNG или Micros |
| `Код` | код объекта во фронт-системе |
| `Объект` | ссылка на объект 1С |
| `НаименованиеКнопки` | текстовое имя объекта фронта |
| `Количество` | количество для строк номенклатуры, если используется |

После записи сопоставления форма запускает в фоне `РегистрыСведений.ПЛ_СоответствиеСФронтСистемами.ЗаполнитьСсылкиПоКодамПослеЗагрузки`, чтобы уже загруженные чеки получили новые ссылки без отдельной перезагрузки.

## Шаг 2. Сверка чеков

Основные процедуры:

- `СтатусШагаСверка(День, МассивСкладов, ПараметрыКасс)`;
- `ПолучитьОтклоненияСверки(День, МассивСкладов, ПараметрыКасс)`;
- `СформироватьОтчетСверкиЧеков(День, МассивСкладов)`;
- `ПерезагрузитьЧекиЗаДень(День, ПараметрыКасс)`.

Проверка сначала убеждается, что за выбранный день есть документы `питДанныеПродажФронта` по выбранным складам и кассам. Если чеков нет, статус шага — `НетДанных`.

Для сверки используются живые источники фронта:

- `MRS_МенеджерОбменаTNG.ВыполнитьПолучениеДанныхTNG(...)`;
- `MRS_МенеджерОбменаMicros.ВыполнитьПолучениеДанныхMicros(...)`.

Результат сравнивается с загруженными чеками 1С. В таблицу отклонений попадают суммы, скидки и категория отклонения. Ошибки вызовов TNG/Micros пишутся в журнал регистрации с событиями `MRS.ПомощникЗакрытияСмены.Сверка.ОшибкаTNG` и `MRS.ПомощникЗакрытияСмены.Сверка.ОшибкаMicros`.

Перезагрузка чеков запускает существующие функции загрузки:

- `MRS_МенеджерОбменаTNG.ВыполнитьЗагрузкуДанныхTNG(...)`;
- `MRS_МенеджерОбменаMicros.ВыполнитьЗагрузкуДанныхMicros(...)`.

Модули обмена TNG/Micros помощник не меняет, только вызывает их публичный интерфейс.

Пользовательский отчет сверки чеков формируется через отчет `ПЛ_СверкаПродажСЧеками`.

## Шаг 3. Формирование целевых документов

Основные процедуры:

- `СтатусШагаФормированиеПоСкладам(...)`;
- `СтатусШагаФормирование(День, Склад, ПараметрыКасс)`;
- `ЧекиЗагруженыПолностью(День, Склад, ПараметрыКасс)`;
- `ЕстьНеактуальныеВыпуски(День, Склад)`;
- `СформироватьЦелевыеДокументы(День, Склад, АдресРезультата)`;
- `ПолучитьОтклоненияВыпуска(День, Склад, ПараметрыКасс)`.

Логика шага идет в таком порядке:

1. Проверить полноту загрузки чеков. Помощник сравнивает номера чеков, которые видит фронт, с номерами документов `питДанныеПродажФронта` в 1С.
2. Проверить актуальность выпусков через `РегистрСведений.питУчетДанныхПродаж`.
3. Если есть неактуальные выпуски, показать пользователю вопрос о пересоздании.
4. Сравнить количество по чекам с количеством по проведенным `питВыпускБлюд`.
5. Показать непроведенные выпуски и документы по открытым ценам.

`питУчетДанныхПродаж` — независимый непериодический регистр сведений. Для помощника важны поля:

| Поле | Смысл |
|---|---|
| `КассовыйУзел` | узел плана обмена `питУдаленныеКассы` |
| `Дата` | дата учета |
| `ДокументРегистратор` | чек `питДанныеПродажФронта` |
| `ДокументПриемник` | целевой документ |
| `ОбновленПриемник` | `Ложь` означает, что целевой документ устарел или не создан |
| `ДатаПоследнегоОбновления` | когда обновлялась связь |

Формирование документов выполняется через существующую обработку `питЗагрузкаИзДанныхПродажФронта`.

Последовательность:

1. `ПометитьЧекиКПересозданию()` выбирает документы-регистраторы из `питУчетДанныхПродаж` по дню и складу приготовления узла, затем перезаписывает чеки `питДанныеПродажФронта`.
2. `ЗаполнитьПараметрыЗагрузкиОбработки()` создает настройки обработки загрузки: день отбора, кассы выбранного склада, склады приготовления/реализации, организацию и подразделение из склада.
3. `ВыполнитьЗагрузкуДанных()` существующей обработки формирует целевые документы.
4. Результат и сообщения пользователя помещаются во временное хранилище для формы.

Основной целевой документ — `Документ.питВыпускБлюд`.

## Шаг 4. Отрицательные остатки

Основные процедуры:

- `СтатусШагаОстатки(День, Склад)`;
- `ПолучитьОтрицательныеОстатки(День, Склад)`;
- `ПолучитьОтрицательныеОстаткиПоСкладам(День, МассивСкладов)`;
- `СформироватьВедомостьОтрицательныхОстатков(...)`;
- `СформироватьАнализОтрицательныхОстатков(...)`.

Источник — регистр накопления `ТоварыНаСкладах`.

Алгоритм:

1. Получить остатки на конец выбранного дня по выбранным складам.
2. Оставить только строки, где `ВНаличииОстаток < 0`.
3. Найти расходные движения этого дня по тем же склад/номенклатура/характеристика.
4. Показать регистратор, который связан с расходом дня.

Статус шага `НеПройден`, если найдена хотя бы одна строка отрицательного остатка.

## Итоговый отчет

`СформироватьИтоговыйОтчет(День, МассивСкладов, ПараметрыКасс)` повторно собирает статусы и возвращает табличный документ.

В отчет попадают:

- выбор складов;
- количество несопоставленных объектов фронта;
- количество отклонений сверки чеков;
- количество отклонений выпусков, ошибок формирования и документов открытых цен;
- количество отрицательных остатков.

## Фоновые операции

Форма запускает тяжелые действия через БСП:

- пересчет статуса шага;
- расчет рекомендуемого дня;
- перезагрузка чеков;
- формирование целевых документов;
- дозаполнение ссылок после сопоставления.

Параметры фонового запуска задаются в `ПараметрыФоновогоЗапуска()`: запуск в фоне включен, расширения базы данных включены, ожидание завершения на сервере не задано. Клиент ждет завершения через `ДлительныеОперацииКлиент.ОжидатьЗавершение` и показывает окно ожидания.

## Карта источников данных

| Данные | Источник | Где используется |
|---|---|---|
| Точки продаж | `КассыККМ` + `питУдаленныеКассы` + `ПЛ_СоответствиеСФронтСистемами` | выбор параметров мастера |
| Загруженные чеки | `Документ.питДанныеПродажФронта` | сопоставление, сверка, полнота загрузки, сверка выпусков |
| Строки чеков | `Документ.питДанныеПродажФронта.Товары` | сопоставление номенклатуры, количество по чекам |
| Данные фронта | `MRS_МенеджерОбменаTNG/ВыполнитьПолучениеДанныхTNG`, `MRS_МенеджерОбменаMicros/ВыполнитьПолучениеДанныхMicros` | сверка чеков и проверка полноты загрузки |
| Загрузка чеков | `MRS_МенеджерОбменаTNG/ВыполнитьЗагрузкуДанныхTNG`, `MRS_МенеджерОбменаMicros/ВыполнитьЗагрузкуДанныхMicros` | действие перезагрузки чеков |
| Сопоставления фронта | `РегистрСведений.ПЛ_СоответствиеСФронтСистемами` | шаг сопоставления и выбор точек продаж |
| Актуальность целевых документов | `РегистрСведений.питУчетДанныхПродаж` | проверка необходимости пересоздания выпусков |
| Формирование выпуска | `Обработка.питЗагрузкаИзДанныхПродажФронта` | создание `питВыпускБлюд` |
| Выпуски блюд | `Документ.питВыпускБлюд` | сверка количества, ошибки формирования, открытые цены |
| Ошибки общепита | `РегистрСведений.ПЛ_ЛогОбщепит` | список непроведенных выпусков и последняя ошибка записи |
| Остатки | `РегистрНакопления.ТоварыНаСкладах` | отрицательные остатки |
| Дата запрета | `РегистрСведений.ДатыЗапретаИзменения` | расчет рекомендуемого дня |
| Отчет сверки чеков | `Отчет.ПЛ_СверкаПродажСЧеками` | табличный отчет на шаге сверки |
| Старая точка входа | `Обработка.MRS_АРМ_Закрытия_смены` | физически остается в расширении, функционально заменяется мастером |

## Статусы

В серверной логике используются статусы:

- `Пройден` — по проверке нет блокирующих проблем;
- `НеПройден` — есть проблемы, которые надо разобрать;
- `НетДанных` — для сверки нет загруженных чеков или не выбраны кассы.

## Технические замечания

- В текущем модуле формы `ПолучитьСтатусШагаПоСтраницеНаСервере()` сразу возвращает `Пройден`, поэтому кнопка **Далее** не является надежной технической блокировкой. Для пользователей важнее фактические таблицы проблем и статус, который отображается после фонового пересчета.
- В `openspec/changes/mrs-shift-closing-assistant/tasks.md` роль `MRS_УдалениеВыпускаБлюд` и часть XML-верификации еще отмечены как незавершенные задачи. При внедрении надо отдельно проверить права на удаление `питВыпускБлюд`.
- Критерий полной загрузки чеков основан на сравнении номеров чеков фронта с номерами документов `питДанныеПродажФронта`. Если во фронте есть незакрытые чеки, они могут не попасть в живой источник фронта и не будут считаться отдельной категорией отклонения.
- Командный интерфейс в этой вычитке не проверялся: точное место пункта меню помощника и скрытие старой обработки надо подтвердить на тестовой базе или через XML командного интерфейса, если он будет доступен в выгрузке.

## Быстрый чеклист для аналитика

1. Точка продаж не видна в мастере — проверить кассу ККМ, `питКассовыйУзел.СкладПриготовления` и запись `ПЛ_СоответствиеСФронтСистемами` для типа `КассаККМ`.
2. На сопоставлении есть строки — проверить коды фронта, записи регистра соответствий и заполненность номенклатуры в `питДанныеПродажФронта.Товары`.
3. На сверке есть отклонения — сравнить живые данные TNG/Micros с `питДанныеПродажФронта`.
4. Формирование не проходит — проверить полноту загрузки, `питУчетДанныхПродаж.ОбновленПриемник`, непроведенные `питВыпускБлюд` и `ПЛ_ЛогОбщепит`.
5. Есть отрицательные остатки — смотреть `ТоварыНаСкладах` на конец дня и регистратор расходного движения.
