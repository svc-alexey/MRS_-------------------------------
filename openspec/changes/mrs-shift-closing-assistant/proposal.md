# Помощник закрытия смены

Пошаговый интерактивный мастер закрытия торгового дня (по образцу типового `Обработки.ПомощникИсправленияОстатковТоваровОрганизаций` — только идея навигации между шагами, не бизнес-логика), заменяющий текущую плоскую MVP-обработку и делающий возможным ежедневное продвижение даты запрета редактирования без последующих правок прошлых периодов.

## Context & Goals

- **Проблема.** Текущий MVP (`DataProcessors/MRS_АРМ_Закрытия_смены`) — одна плоская форма без понятия «шаг» и без блокировки перехода вперёд при непройденном шаге. Пользователь может (по невнимательности) двинуть дату запрета, не проверив сверку или отрицательные остатки, и потом эти правки задним числом становится дорого/невозможно провести аккуратно.
- **Фон.** Точки продаж торгуют на кассах Micros/TNG; ночью грузятся чеки предыдущего дня; строковые объекты фронта (кассы, номенклатура) сопоставляются с объектами 1С регистром `РегистрСведений.ПЛ_СоответствиеСФронтСистемами`; загруженные чеки сверяются с фронтом; из чеков формируются целевые документы производства (общепит, 1С:ERP + модуль «Общепит» Рарус); после производства проверяются отрицательные остатки. Ответственный за склад должен закрыть день без отрицательных остатков, чтобы можно было безопасно двигать дату запрета редактирования вперёд.
- **Объективы.**
  - Провести пользователя по фиксированной последовательности шагов (сопоставление → сверка → формирование целевых документов → анализ отрицательных остатков) с явной блокировкой перехода вперёд без завершения текущего шага. Загрузка чеков за день выполняется автоматически фоновым заданием до открытия мастера и не входит в эту последовательность как отдельный шаг.
  - Дать пользователю выбор дня и одного/нескольких складов на каждый запуск мастера и подсветить, с какого дня рекомендуется начинать.
  - Заменить плоский, перегруженный для пользователя отчёт сверки (`Reports/MRS_ПродажиФронтСистем`) на шаг, показывающий либо конкретные строки отклонений (с суммами по фронту и по 1С), либо явное «всё чисто».
  - Переосмыслить UX сопоставления несопоставленных объектов фронта (кассы и номенклатура через `РегистрСведений.ПЛ_СоответствиеСФронтСистемами`) — текущая форма регистра признана неудобной для этого сценария.
  - Сохранить как есть (использовать «чёрным ящиком») существующие механизмы загрузки чеков (`ОбщийМодуль.MRS_МенеджерОбменаTNG`, `ОбщийМодуль.MRS_МенеджерОбменаMicros`) и формирования целевых документов производства (`DataProcessor.питЗагрузкаИзДанныхПродажФронта` — имя объекта и программный интерфейс `ВыполнитьЗагрузкуДанных()` подтверждены чтением кода).
  - Там, где БСП предоставляет готовый механизм (длительные операции, стандартные пользовательские сообщения/диалоги и т.п.) — предпочитать его перед собственной реализацией с нуля; конкретный перечень применяемых БСП-подсистем и API — решение `design.md` (`1c-architect`), это proposal фиксирует только сам принцип предпочтения.
- **Non-goals / Out of scope.**
  - Доработка или редизайн бизнес-логики загрузки чеков внутри `MRS_МенеджерОбменаTNG`/`MRS_МенеджерОбменаMicros` — мастер только запускает существующий механизм и показывает результат.
  - Доработка или редизайн документа `Документ.питРецептура` (состав рецептуры, состав производственных подразделений) — используется как есть.
  - Доработка или редизайн существующей обработки `DataProcessor.питЗагрузкаИзДанныхПродажФронта`, формирующей целевые документы («Выпуск блюд» и прочие движения не по чекам) из чеков — мастер только инициирует её запуск (программный интерфейс — экспортная функция `ВыполнитьЗагрузкуДанных()`) и показывает её результат/ошибки.
  - Доработка или редизайн `Документ.питАктВскрытияТарыАлкогольнойПродукции` и регистра `РегистрНакопления.питОстаткиМарокИС` — мастер только проверяет достаточность остатков марок для производства и подсказывает пользователю, что нужно оформить акт (не создаёт и не проводит акт от имени пользователя).
  - Запись/изменение стандартного `РегистрСведений.ДатыЗапретаИзменения` — продвижение даты запрета редактирования остаётся внешней типовой операцией ERP; мастер только показывает готовность («день чист по выбранным складам, можно двигать дату») и не пишет в этот регистр.
  - Автоматический откат статуса шага/дня при повторном сопоставлении номенклатуры или повторной загрузке чека после того, как пользователь прошёл дальше по шагам — пользователь возвращается к нужному шагу вручную (см. `Требование: Отсутствие автоматического отката статуса шага` в domain оркестрации).
  - Интерактивные исправления отрицательных остатков прямо в мастере (создание перемещений/списаний/правка рецептуры из шага анализа) — см. `## Open Questions`.
  - Ролевая/полномочная модель по складам/точкам продаж — не вводится: единая роль «ответственный», без привязки к конкретным складам на уровне прав или данных.
  - Проектирование конкретного состава новой повышенной роли для удаления `питВыпускБлюд` (имя, состав объектов и прав) — решение `design.md`/`1c-metadata-manager`; это proposal фиксирует только сам факт необходимости ограничения (см. `## Core Functions`, `## Metadata`).

## Affected users

- **Ответственный за склад/точку продаж** — единая роль, без привязки к конкретным складам/точкам продаж на уровне прав или данных. На каждый запуск мастера пользователь сам выбирает, с какими складами работает в рамках сессии (из всех доступных в информационной базе).

## Core Functions

- Выбор дня и набора складов на сессию мастера, с подсветкой рекомендуемого дня начала.
- Жёсткая пошаговая навигация: Сопоставление объектов фронта → Сверка чеков → Формирование целевых документов → Анализ отрицательных остатков.
- Явный статус каждого шага для выбранного дня/складов (пройден / не пройден / недоступен), вычисляемый по фактическому состоянию данных (загруженные чеки, несопоставленные объекты, отклонения сверки, непроведённые/несформированные документы, отрицательные остатки) — не по отдельному ручному чек-боксу пользователя.
- (Пере)загрузка чеков за день и переформирование выпусков — точечные действия, доступные прямо на соответствующих шагах (перезагрузка — на шагах «Сверка чеков» и «Сопоставление объектов фронта»; переформирование — на шаге «Формирование целевых документов»), а не отдельные шаги мастера.
- Переосмысленная форма сопоставления несопоставленных объектов фронта (кассы и номенклатура) поверх `РегистрСведений.ПЛ_СоответствиеСФронтСистемами`.
- Явный вывод отклонений сверки чеков (строками, с суммами по фронту и по 1С) либо явное «всё чисто».
- Проверка актуальности выпусков при входе на шаг «Формирование целевых документов»: мастер проверяет `РегистрСведений.питУчетДанныхПродаж.ОбновленПриемник` по чекам выбранного дня/складов (подтверждённый кодом регистр расширения — `ObjectBelonging=Adopted`, не референс базовой конфигурации) и, если найдены неактуальные записи, показывает пользователю явную рекомендацию с подтверждением («Рекомендуется пересоздание выпусков блюд для обновления. Выполнить сейчас? Да/Нет») вместо тихого автоматического запуска. «Нет» — шаг остаётся на паузе (не пройден), без блокировки остального мастера. «Да» — запускает существующую обработку `DataProcessor.питЗагрузкаИзДанныхПродажФронта` (программный интерфейс — экспортная функция `ВыполнитьЗагрузкуДанных()`) в фоне (БСП `ДлительныеОперации`), строго с отбором по выбранным пользователем складам и дате — без изменения `MRS_МенеджерОбменаTNG`/`MRS_МенеджерОбменаMicros`, которые в этой механике вообще не участвуют.
- Предусловие полноты загрузки: генерация/подсказка недоступна, если чеки за выбранный день загружены не полностью — к созданию выпуска должны быть учтены все чеки дня, а не часть.
- Действие «Переформировать выпуска» — тот же механизм генерации, доступный пользователю в любой момент на этом шаге по явному действию, а не только по автоматической подсказке при входе.
- Явный вывод ошибок формирования обработки (нехватка марок/акта вскрытия тары, нехватка ингредиентов, отрицательные остатки после выпуска, прочие ошибки непроведения), а после успешной генерации — показ созданного документа выпуска пользователю через существующий шаблон печатной формы обработки `питЗагрузкаИзДанныхПродажФронта` (переиспользуется как есть).
- Сверка чеков и выпусков: сравнение агрегированного за день по складу количества по номенклатуре между чеками и проведёнными документами `питВыпускБлюд` — отрицательное отклонение блокирует шаг, положительное показывается как информационная пометка (не блокирует).
- Отдельный явный список документов, сформированных по «открытым ценам» и требующих дозаполнения конкретной номенклатурой пользователем.
- Ограничение права удаления документа `питВыпускБлюд` отдельной повышенной ролью (либо ролью «Полные права») — обычный пользователь мастера этой возможности не имеет.
- Финальный шаг анализа отрицательных остатков с явным «всё чисто» либо перечнем строк с остатком < 0.
- Готовность к продвижению даты запрета редактирования — как индикация, без записи в стандартный регистр.

## Flows (Text-Only)

1. Пользователь открывает мастер → выбирает день (мастер подсказывает рекомендуемый — самый ранний день среди доступных складов, где хотя бы один шаг не пройден) и один либо несколько складов.
2. Мастер вычисляет и показывает статус каждого шага для выбранной комбинации день+склады. Первый непройденный шаг открыт для работы, шаги после него недоступны для перехода (только просмотр статуса, без перехода к работе внутри них).
3. К этому моменту чеки за выбранный день, как правило, уже загружены автоматически фоновым заданием (`MRS_ЗагрузкаЧековTNG`/`MRS_ЗагрузкаЧековMicros`) до открытия мастера, без участия пользователя. Отдельного шага мастера для инициации загрузки нет.
4. **Шаг «Сопоставление объектов фронта».** Мастер показывает список несопоставленных касс/точек продаж и несопоставленной номенклатуры фронта, встреченных в загруженных за день чеках, в новой форме поверх `ПЛ_СоответствиеСФронтСистемами`. Шаг пройден, когда для всех объектов фронта, встреченных в чеках выбранного дня, есть сопоставление. На этом шаге доступно точечное действие «(Пере)загрузить чеки за день» — чтобы после донастройки сопоставления новые объекты подтянулись в данные.
5. **Шаг «Сверка чеков».** Мастер сравнивает загруженные в 1С данные с данными фронта и показывает построчно отклонения (с суммами/скидками по фронту и по 1С) либо «всё чисто». Шаг пройден только при отсутствии отклонений (любое ненулевое отклонение блокирует переход). Чеки, не загруженные из-за незакрытости на фронте на момент загрузки, не считаются отдельной категорией отклонений (они просто отсутствуют в загруженных данных на этот момент). Если данных за день нет вовсе (автозагрузка не выполнялась или упала с ошибкой) либо есть отклонения, объяснимые более поздним изменением/закрытием чека на фронте, пользователю доступно точечное действие «(Пере)загрузить чеки за день» прямо на этом шаге — оно закрывает оба случая без отдельного «шага загрузки».
6. **Шаг «Формирование целевых документов».** При входе на шаг мастер проверяет `РегистрСведений.питУчетДанныхПродаж.ОбновленПриемник` по чекам выбранного дня и складов. Если чеки за день загружены НЕ полностью — генерация недоступна, шаг явно показывает предупреждение о неполной загрузке вместо предложения сгенерировать. Если данные полны, но найдены записи с `ОбновленПриемник = Ложь` (целевой документ устарел относительно чека — например, из-за (пере)загрузки чеков или донастройки сопоставления на предыдущих шагах) — мастер показывает рекомендацию «Рекомендуется пересоздание выпусков блюд для обновления. Выполнить сейчас?» с явным подтверждением Да/Нет. При «Нет» шаг остаётся на паузе (не пройден), пользователь может покинуть мастер и вернуться позже. При «Да» — мастер помечает соответствующие чеки к пересозданию (переиспользуя существующий паттерн модуля «Общепит» — поиск чеков с `ОбновленПриемник ≠ Истина` в границах день+склады и их перезапись) и запускает в фоне (БСП `ДлительныеОперации`) обработку `DataProcessor.питЗагрузкаИзДанныхПродажФронта` (программный интерфейс — экспортная функция `ВыполнитьЗагрузкуДанных()`) строго с отбором по выбранным складам и дате — без изменения `MRS_МенеджерОбменаTNG`/`MRS_МенеджерОбменаMicros`. То же самое действие доступно и вручную в любой момент на этом шаге через точечное действие «Переформировать выпуска» (независимо от того, обнаружила ли автоматическая проверка неактуальность). После успешной генерации мастер показывает пользователю созданный документ `питВыпускБлюд` через существующий шаблон печатной формы обработки (переиспользуется как есть) с ошибками проведения, если они есть. Шаг также показывает: список ошибок непроведения (нехватка акта вскрытия тары / марок, нехватка ингредиентов, отрицательные остатки, прочие ошибки — при наличии); отдельно — список документов по «открытым ценам», ожидающих дозаполнения конкретной номенклатурой и проведения; и результат сверки чеков и выпусков — сравнения агрегированного за день по складу количества по номенклатуре между чеками и проведёнными документами `питВыпускБлюд`. Отрицательное отклонение (в выпуске меньше, чем по чекам) показывается как строка проблемы и блокирует шаг; положительное отклонение (в выпуске больше — пользователь мог сам добавить строки в документ вручную) показывается как информационная пометка и не блокирует шаг. Шаг пройден, когда для выбранных складов и дня все ожидаемые документы (в т.ч. по открытым ценам) проведены, нет необработанных ошибок формирования и нет отрицательных отклонений сверки чеков и выпусков. Удаление документа `питВыпускБлюд` доступно только пользователю с отдельной повышенной ролью (или ролью «Полные права») — обычный пользователь мастера этой возможности не имеет.
7. **Шаг «Анализ отрицательных остатков».** Мастер показывает отчёт по остаткам < 0 по выбранным складам/дню либо «всё чисто». Шаг пройден при отсутствии отрицательных остатков. Исправление остатков в этой версии выполняется пользователем вне мастера (см. `## Open Questions`); после исправления пользователь возвращается на этот шаг для повторной проверки.
8. Когда все шаги для выбранной комбинации день+склады пройдены — мастер показывает готовность («день чист по выбранным складам, дату запрета редактирования можно продвигать»); фактическое продвижение даты пользователь выполняет типовыми средствами ERP вне мастера.
9. Если складов, выбранных на сессию, для этого дня несколько — пункты 4-7 проходятся для комбинации в целом (сопоставление/сверка агрегированы по дню; формирование документов и анализ остатков — по каждому складу, так как эти шаги складо-специфичны).
10. После завершения по выбранной комбинации пользователь может выбрать следующий проблемный день/склады (мастер вновь подсвечивает рекомендуемый день среди оставшихся непройденных).

## Data & Integrations

- **Дни/склады со статусом «не закрыт».** Определяются на лету по фактическому состоянию данных (нет персистентного объекта статуса шага, вводимого этим change) — критерии по шагам см. `## Metadata` и delta-спеки. Экземпляр запуска мастера не хранит состояние между сессиями сверх того, что уже отражено в прикладных данных (чеки, документы, отклонения).
- **Загрузка чеков** — как правило выполняется автоматически до открытия мастера существующими фоновыми заданиями `MRS_ЗагрузкаЧековTNG` / `MRS_ЗагрузкаЧековMicros`. Мастер добавляет точечное действие «(Пере)загрузить чеки за день» на шагах «Сверка чеков» и «Сопоставление объектов фронта», вызывающее те же `ОбщийМодуль.MRS_МенеджерОбменаTNG` / `ОбщийМодуль.MRS_МенеджерОбменаMicros`, без изменения их внутренней логики.
- **Сопоставление объектов фронта** — читает/пишет `РегистрСведений.ПЛ_СоответствиеСФронтСистемами` (существующий регистр модуля «Общепит», обслуживающий и кассы, и номенклатуру через дискриминатор типа объекта); новая форма — единственный новый UI-артефакт в этом домене.
- **Сверка чеков** — сравнивает `Документ.питДанныеПродажФронта` (загруженные в 1С чеки) с исходными данными фронта (то, что уже сравнивается в существующем СКД-отчёте `Reports/MRS_ПродажиФронтСистем` — источник данных может быть переиспользован на уровне design.md, но не сам отчёт как пользовательский UI).
- **Формирование целевых документов** — интерактивный механизм на шаге мастера (не тихий автозапуск): актуальность проверяется через `РегистрСведений.питУчетДанныхПродаж.ОбновленПриемник` (Adopted-регистр расширения — измерения `КассовыйУзел`, `Дата`, `ДокументРегистратор` [чек `питДанныеПродажФронта`], `ДокументПриемник` [полиморфная ссылка на целевой документ], `ОбновленПриемник` [Булево]); при обнаружении неактуальных записей в границах выбранных день+склады — явный запрос подтверждения у пользователя, затем фоновый (БСП `ДлительныеОперации`) запуск программного интерфейса `DataProcessor.питЗагрузкаИзДанныхПродажФронта.ВыполнитьЗагрузкуДанных()` (подтверждён чтением кода), строго с отбором по выбранным складам/дате; результат — документы `Документ.питВыпускБлюд` (в т.ч. с признаком происхождения `ПЛ_Номенклатура` для документов «открытых цен») и, для складов без выпуска по чекам, прочие документы (перемещения/списания), формируемые той же обработкой.
- **Проверка обеспеченности алкоголем** — проверка наличия оприходованных по `Документ.питАктВскрытияТарыАлкогольнойПродукции` марок в остатках `РегистрНакопления.питОстаткиМарокИС` (измерения `Организация`+`МестоПечати` [фактически хранит склад — подтверждено кодом `MRS_ЗагрузкаЕГАИСФронт`] +`Номенклатура`+`Штрихкод`, ресурс `Количество`), достаточных для производства блюд, использующих алкоголь как ингредиент, по рецептуре из `Документ.питРецептура`.
- **Анализ отрицательных остатков** — без изменений использует существующий механизм анализа остатков (текущий MVP использует `АнализДанных`/`ПостроительОтчетаАнализаДанных` по регистру накопления остатков товаров; конкретный источник — решение design.md).

## Assumptions

- «День» — календарная дата (`НАЧАЛОПЕРИОДА(..., ДЕНЬ)`), единая для загрузки/сопоставления/сверки; для формирования документов и анализа остатков — та же дата в разрезе каждого выбранного склада.
- Набор доступных для выбора складов — все `Справочник.Склады`, зарегистрированные в системе (единая ролевая модель, без фильтрации по правам пользователя на конкретный склад).
- Обработка `питЗагрузкаИзДанныхПродажФронта` и документ `питАктВскрытияТарыАлкогольнойПродукции`/регистр `питОстаткиМарокИС` уже поддерживают необходимую бизнес-логику (списание марок по ФИФО, привязку к производственному подразделению через `питРецептура`) без доработок — сам факт наличия этой логики подтверждён пользователем, не вычитан из кода построчно (структура объектов — да, подтверждена чтением кода).
- Существующий СКД-отчёт `Reports/MRS_ПродажиФронтСистем` (схемы TNG и Micros) остаётся технически доступным как источник данных для шага сверки на уровне запроса/выборки — сам отчёт как пользовательская форма выводится из использования, но его СКД-схемы не обязательно удалять физически (решение — `design.md`).

## Risks & Mitigations

| Риск | Влияние | Митигация |
|---|---|---|
Точный критерий «чеки за день загружены полностью» (предусловие генерации) не определён фактами этой сессии — доступные объекты (`питУчетДанныхПродаж`, `ПЛ_СтатусыОбработкиЧековОбщепита`) отслеживают актуальность/обработку отдельных чеков, но не дают готового флага «за день N загружены ВСЕ чеки» | Без надёжного критерия мастер может либо ложно блокировать генерацию (день на самом деле полон), либо ложно её разрешать (день по факту неполон, выпуски получатся по неполным данным) | Уточняется на этапе `design.md`/`1c-architect` — вероятные источники: сопоставление с фактическим количеством чеков на фронте (`ВыполнитьПолучениеДанныхTNG`/`Micros`, уже используется шагом сверки) либо отдельный флаг «смена/день закрыта на фронте», если такой существует в основной конфигурации |
| Мастер помечает чеки к пересозданию через `Записать()` документа `питДанныеПродажФронта` (паттерн `УстановитьВозможностьЗагрузкиНаСервере`) — внутренняя механика того, что именно происходит при этой записи (что именно взводит `ОбновленПриемник=Ложь`/запускает пересоздание приёмника), не вычитана построчно в этой сессии, зафиксирована как чёрный ящик | Если реальная механика сложнее простой перезаписи (например, зависит от дополнительных условий или скрытых подписок на событие в основной конфигурации), фактическое поведение мастера на этом шаге может отличаться от ожидаемого | Уточняется на этапе `design.md`/реализации — проверяется на тестовой ИБ перед вводом в эксплуатацию |
| Отсутствие персистентного статуса шага означает, что статус всегда вычисляется запросом по актуальным данным — на больших объёмах чеков/документов это может быть медленно | Задержка отклика мастера при открытии для дня с большим количеством чеков | Производительность запросов — предмет `design.md`/`1c-performance-optimizer`, не заявлена как ограничение на уровне требований |
| Пользователь путает «День чист, дату можно двигать» (индикация мастера) с фактическим состоянием `ДатыЗапретаИзменения` (внешняя операция) | Ложное ощущение завершённости, если стандартную дату не продвинули | Формулировка индикации явно поясняет, что это подсказка, а не факт продвижения даты (см. delta-спеку оркестрации) |

## Success Criteria

- Ответственный закрывает выбранный день по выбранным складам без отрицательных остатков за один проход мастера (без обращения к трём разным объектам — старой обработке, старому отчёту сверки, стандартному анализу остатков — по отдельности).
- Ноль незамеченных отклонений сверки: каждое ненулевое отклонение по сумме/скидке видно пользователю построчно с суммами по фронту и по 1С, а не только в виде общего перегруженного отчёта.
- Мастер не позволяет перейти к формированию целевых документов/анализу остатков, если сопоставление или сверка не завершены (0 случаев обхода шагов не по кнопке «назад»/повторного запуска шага).

## Open Questions

1. **Интерактивные исправления отрицательных остатков внутри мастера.** Вопрос: должен ли шаг «Анализ отрицательных остатков» (домен `smena-negative-stock-resolution`) в будущей версии поддерживать исполнение исправляющих действий (создание перемещения, инициирование правки рецептуры и т.п.) прямо из мастера, или он останется чисто информационным (`report_only`)? Варианты: (A) остаётся `report_only` — пользователь исправляет вне мастера, шаг только перепроверяет; (B) добавляются конкретные интерактивные действия — но конкретный список автоматизируемых сценариев исправления пока не определён (причины отрицательных остатков разнородны: рецептура, отсутствие замены, отсутствие перемещения). Не может быть закрыт сейчас — зависит от будущего решения о том, какие сценарии исправления целесообразно автоматизировать. Затрагиваемый раздел: `specs/smena-negative-stock-resolution/spec.md → Requirement: Отображение отрицательных остатков`. Зависимая задача: любая задача `tasks.md`, реализующая действия на этом шаге сверх отображения отчёта (в текущей версии таких задач нет — шаг ограничен отображением).

Пункт про точный алгоритм расчёта ожидаемого по чекам количества для сверки чеков и выпусков **закрыт**: образцовый отчёт `Reports/ПЛ_СверкаПродажСЧеками` (СКД-схема, предоставлена пользователем) подтвердил принцип — простое совпадение количества, агрегированное по номенклатуре (с особым случаем для характеристики при `питПродажаПоСвободнойЦене`) + день + склад, БЕЗ пересчёта через рецептурные коэффициенты. Перенесено в `specs/smena-target-document-generation/spec.md → Requirement: Сверка чеков и выпусков`.

Отдельную БСП-подсистему навигации по шагам/мастерам `ssl_search` не подтвердил (два запроса с разными формулировками вернули нерелевантные результаты — ближайшие совпадения из подсистем «Обсуждения», «Контактная информация», «Отправка SMS»), поэтому такой паттерн не фиксируется как факт; решение по механике навигации по шагам — `design.md`. Пользователь явно подтвердил: использование БСП для нового мастера — строго желательно, не просто допустимо.

Требование к правам: удаление `питВыпускБлюд` — только с отдельной повышенной ролью (или «Полные права», стандартный факт платформы).
