# Delta for smena-front-object-matching

## ADDED Requirements

### Requirement: Единая точка сопоставления касс и номенклатуры фронта
Шаг «Сопоставление объектов фронта» ДОЛЖЕН показывать в одном экране несопоставленные объекты обоих типов, зарегистрированных в `РегистрСведений.ПЛ_СоответствиеСФронтСистемами` через дискриминатор `ТипОбъекта`: несопоставленные кассы/точки продаж и несопоставленную номенклатуру, встреченные в чеках выбранного дня.

#### Scenario: В чеках дня встречены и несопоставленная касса, и несопоставленная номенклатура
- GIVEN в загруженных за 2026-07-05 чеках Micros встречена точка продаж, для которой нет записи в `ПЛ_СоответствиеСФронтСистемами` с `ТипОбъекта` = касса, и код товара, для которого нет записи с `ТипОбъекта` = номенклатура
- WHEN пользователь открывает шаг «Сопоставление объектов фронта»
- THEN мастер показывает оба несопоставленных объекта в одном списке (или в двух явно озаглавленных секциях одной формы), с указанием кода/наименования из фронта и фронт-системы

### Requirement: Новая форма сопоставления взамен типовой формы регистра
Шаг «Сопоставление объектов фронта» ДОЛЖЕН использовать новую форму, спроектированную для этого сценария (позволяющую выбрать объект 1С для сопоставления прямо из списка несопоставленных, без перехода в стандартный список регистра `ПЛ_СоответствиеСФронтСистемами`), а НЕ типовую форму списка/элемента этого регистра. Подтверждено чтением кода существующей `Forms/ФормаЗаписи`: она рассчитана на редактирование ОДНОЙ записи регистра (поля `ТипОбъекта`/`Объект`/`Количество`/`Упаковка`/`НоменклатураАлкоголь`, видимость которых переключается динамически в зависимости от выбранного типа объекта) — открытие, заполнение и запись каждой записи по отдельности через стандартную навигацию списка регистра, без представления сразу списка несопоставленных объектов дня и без пакетного сопоставления.

#### Scenario: Сопоставление кассы без перехода в список регистра
- GIVEN пользователь видит в шаге сопоставления несопоставленную кассу с кодом точки продаж «102»
- WHEN пользователь выбирает существующую кассу `Справочник.КассыККМ` для сопоставления и подтверждает выбор
- THEN мастер создаёт соответствующую запись в `ПЛ_СоответствиеСФронтСистемами` (Объект/ФронтСистема/Код/ТипОбъекта = касса) без открытия типовой формы списка или элемента регистра

### Requirement: Автоматическое дозаполнение ссылок в загруженных чеках после сопоставления
Сразу после того как пользователь сопоставил объект фронта (кассу или номенклатуру) и подтвердил это действие («Сопоставить»/пакетное подтверждение), мастер ДОЛЖЕН программно вызвать существующую экспортную точку менеджера регистра `РегистрыСведений.ПЛ_СоответствиеСФронтСистемами.ЗаполнитьСсылкиПоКодамПослеЗагрузки` (синхронно) либо её фоновую обёртку `ЗаполнитьСсылкиПоКодамПослеЗагрузкиВФоне` (через БСП `ДлительныеОперации`) — программный интерфейс подтверждён чтением кода `Ext/ManagerModule.bsl`. Вызов ДОЛЖЕН дозаполнять ссылку на сопоставленный объект во всех уже загруженных чеках, где сопоставление по этому коду ещё не завершено (внутренний фильтр обработки — `НЕ СопоставлениеЗавершено И НЕ ПометкаУдаления И Дата > ДатаЗапрета`, без параметра «день»), без необходимости для пользователя выполнять отдельное действие «(Пере)загрузить чеки за день» для этой цели.

#### Scenario: Дозаполнение чеков сразу после сопоставления кассы
- GIVEN за 2026-07-05 и предыдущие несколько дней в чеках Micros встречается точка продаж с кодом «102», для которой в `ПЛ_СоответствиеСФронтСистемами` нет сопоставления (`Объект` не заполнен), и по этим чекам сопоставление ещё не завершено
- WHEN пользователь на шаге «Сопоставление объектов фронта» выбирает для кода «102» кассу `Справочник.КассыККМ.Касса-Бар1` и подтверждает сопоставление
- THEN мастер записывает соответствие в `ПЛ_СоответствиеСФронтСистемами`
- AND сразу вызывает `ЗаполнитьСсылкиПоКодамПослеЗагрузки[ВФоне]`, в результате чего все ранее загруженные чеки с кодом «102» получают заполненную ссылку на «Касса-Бар1»
- AND код «102» покидает список несопоставленных объектов на этом шаге без отдельной перезагрузки чеков

#### Scenario: Дозаполнение не требует указания дня
- GIVEN пользователь сопоставил номенклатуру, встречавшуюся в чеках за несколько разных дней (в т.ч. до выбранного в сессии мастера дня)
- WHEN мастер вызывает `ЗаполнитьСсылкиПоКодамПослеЗагрузки` после сопоставления
- THEN вызов выполняется без параметра дня/периода — обрабатываются все чеки с этим кодом, где сопоставление ещё не завершено и которые находятся после даты запрета редактирования, независимо от выбранного в сессии мастера дня

### Requirement: Блокировка формирования документов учёта по несопоставленным объектам
Пока объект фронта (касса или номенклатура), встреченный в чеках выбранного дня, остаётся несопоставленным, шаг «Сопоставление объектов фронта» СЧИТАЕТСЯ не пройденным, и последующие шаги мастера (Сверка чеков, Формирование целевых документов) НЕ ДОЛЖНЫ становиться доступными для этого дня.

#### Scenario: Хотя бы один объект не сопоставлен
- GIVEN за 2026-07-05 в чеках встречено 5 уникальных кодов номенклатуры, 4 из них сопоставлены, 1 — нет
- WHEN мастер вычисляет статус шага «Сопоставление объектов фронта»
- THEN шаг отмечается как не пройденный
- AND переход к шагу «Сверка чеков» для 2026-07-05 недоступен

#### Scenario: Все объекты дня сопоставлены
- GIVEN все объекты фронта (кассы и номенклатура), встреченные в чеках за 2026-07-05, сопоставлены записями в `ПЛ_СоответствиеСФронтСистемами`
- WHEN мастер вычисляет статус шага «Сопоставление объектов фронта»
- THEN шаг отмечается как пройденный
