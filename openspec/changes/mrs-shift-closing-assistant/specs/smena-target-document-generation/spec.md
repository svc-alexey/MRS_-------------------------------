# Delta for smena-target-document-generation

## ADDED Requirements

### Requirement: Проверка актуальности выпусков и предложение пересоздания при входе на шаг
При открытии шага «Формирование целевых документов» для работы мастер ДОЛЖЕН проверять актуальность целевых документов относительно чеков выбранного дня и складов через `РегистрСведений.питУчетДанныхПродаж.ОбновленПриемник` (Adopted-регистр расширения `MRS_ЗакрытиеСмены`; измерения `ДокументРегистратор` [чек `Документ.питДанныеПродажФронта`], `ДокументПриемник` [целевой документ], `ОбновленПриемник` [Булево — Ложь означает, что целевой документ устарел относительно чека или не создан]). Если в границах выбранных день+склады найдена хотя бы одна запись с `ОбновленПриемник = Ложь`, мастер ДОЛЖЕН показать пользователю явную рекомендацию с запросом подтверждения: «Рекомендуется пересоздание выпусков блюд для обновления. Выполнить сейчас?» (Да/Нет).
- При ответе «Нет» — шаг остаётся на паузе (не пройден); мастер НЕ ДОЛЖЕН блокировать остальную работу — пользователь может покинуть шаг/мастер и вернуться позже, статус будет пересчитан заново при следующем открытии шага.
- При ответе «Да» — мастер выполняет действие «Переформировать выпуска» (см. ниже) программно, с отбором строго по выбранным день+склады.

Тихий автоматический запуск без участия пользователя (в т.ч. из фонового задания загрузки чеков) НЕ ДОЛЖЕН применяться — предыдущая формулировка этого требования, предполагавшая автозапуск сразу после загрузки чеков, ОТМЕНЕНА пользователем и заменена этим интерактивным механизмом.

#### Scenario: Обнаружена неактуальность — пользователь подтверждает пересоздание
- GIVEN пользователь на шаге «Сверка чеков» выполнил действие «(Пере)загрузить чеки за день» за 2026-07-05 по складу «Кухня-1», из-за чего в `питУчетДанныхПродаж` для чеков «Кухня-1» за 2026-07-05 `ОбновленПриемник = Ложь`
- WHEN пользователь открывает шаг «Формирование целевых документов» за 2026-07-05 для «Кухня-1»
- THEN мастер показывает рекомендацию «Рекомендуется пересоздание выпусков блюд для обновления. Выполнить сейчас?» с вариантами Да/Нет

#### Scenario: Пользователь отказывается — шаг остаётся на паузе
- GIVEN мастер показал рекомендацию пересоздания на шаге «Формирование целевых документов»
- WHEN пользователь выбирает «Нет»
- THEN шаг остаётся не пройденным, но пользователь может продолжить работу с другими шагами/склад-днями сессии без блокировки
- AND при повторном открытии этого шага рекомендация показывается снова, пока данные остаются неактуальными

#### Scenario: Все выпуски актуальны — рекомендация не показывается
- GIVEN для выбранного дня и складов все записи `питУчетДанныхПродаж` в границах отбора имеют `ОбновленПриемник = Истина`
- WHEN пользователь открывает шаг «Формирование целевых документов»
- THEN мастер не показывает рекомендацию пересоздания и переходит к показу текущего результата (списка ошибок, отклонений сверки чеков и выпусков, документов по открытым ценам)

### Requirement: Предусловие полноты загрузки чеков за день
Действие генерации (автоматическая рекомендация с подтверждением и ручное «Переформировать выпуска») НЕ ДОЛЖНО быть доступно, если чеки фронт-системы за выбранный день загружены НЕ полностью — к созданию выпуска должны быть учтены все чеки дня, а не только часть. Пока это условие не выполнено, мастер ДОЛЖЕН показать пользователю явное предупреждение о неполной загрузке вместо предложения сгенерировать. Точный критерий «загружены полностью» в этом change НЕ фиксируется (см. `../../proposal.md → ## Risks & Mitigations`) — решение `design.md`.

#### Scenario: Частичная загрузка блокирует генерацию
- GIVEN за 2026-07-05 по складу «Кухня-1» загружена только часть чеков дня (загрузка не завершена/прервана)
- WHEN пользователь открывает шаг «Формирование целевых документов» за 2026-07-05 для «Кухня-1»
- THEN мастер показывает предупреждение о неполной загрузке чеков за день вместо рекомендации пересоздания или доступного действия «Переформировать выпуска»

### Requirement: Действие «Переформировать выпуска»
Шаг «Формирование целевых документов» ДОЛЖЕН предоставлять пользователю точечное действие «Переформировать выпуска», доступное в любой момент на этом шаге (независимо от того, обнаружила ли автоматическая проверка неактуальность). Действие ДОЛЖНО: (а) пометить к пересозданию чеки выбранного дня/складов, у которых `питУчетДанныхПродаж.ОбновленПриемник ≠ Истина` — переиспользуя существующий в модуле «Общепит» паттерн (подтверждён чтением кода `DataProcessors/питЗагрузкаИзДанныхПродажФронта/Forms/ВозможностьЗагрузкиДПФ/Ext/Form/Module.bsl`: поиск чеков по регистру `питУчетДанныхПродаж` в границах фильтра и их перезапись, `MRS_ОбновитьСписокНаСервере`/`УстановитьВозможностьЗагрузкиНаСервере`) — точная реализация в общем модуле мастера (реюз этих процедур либо эквивалентная логика) НЕ фиксируется в этом change, решение `design.md`; (б) запустить в фоне (БСП `ДлительныеОперации`) программный интерфейс `DataProcessor.питЗагрузкаИзДанныхПродажФронта.ВыполнитьЗагрузкуДанных()`, строго с отбором по выбранным пользователем складам и дате (не по всей организации); (в) после успешного завершения показать пользователю созданный документ `Документ.питВыпускБлюд` через существующий шаблон печатной формы обработки `питЗагрузкаИзДанныхПродажФронта` (переиспользуется как есть, со слов пользователя — не подтверждено чтением файла шаблона в этой сессии) с ошибками проведения, если они есть.

#### Scenario: Переформирование по явному действию пользователя
- GIVEN пользователь ранее сформировал целевые документы за 2026-07-05 по складу «Кухня-1», затем на шаге «Сверка чеков» выполнил действие «(Пере)загрузить чеки за день» за 2026-07-05
- WHEN пользователь возвращается на шаг «Формирование целевых документов» и запускает действие «Переформировать выпуска»
- THEN мастер помечает к пересозданию чеки «Кухня-1» за 2026-07-05 с неактуальным приёмником, запускает в фоне `питЗагрузкаИзДанныхПродажФронта.ВыполнитьЗагрузкуДанных()` строго по «Кухня-1»/2026-07-05
- AND после завершения показывает созданный документ выпуска через шаблон печатной формы обработки, включая ошибки проведения, если они есть

#### Scenario: Переформирование запущено из подтверждения автоматической рекомендации
- GIVEN мастер показал рекомендацию пересоздания на шаге «Формирование целевых документов» за 2026-07-05 для «Кухня-1»
- WHEN пользователь отвечает «Да»
- THEN мастер выполняет то же действие «Переформировать выпуска» программно, с тем же отбором по «Кухня-1»/2026-07-05

### Requirement: Сверка чеков и выпусков
Шаг «Формирование целевых документов» ДОЛЖЕН сравнивать агрегированное за день по складу количество по номенклатуре, ожидаемое по чекам (`Документ.питДанныеПродажФронта`), с фактическим количеством по той же номенклатуре в проведённых документах `Документ.питВыпускБлюд` за тот же день и склад. Принцип агрегации подтверждён образцовым отчётом сверки `Reports/ПЛ_СверкаПродажСЧеками` (СКД-схема, предоставлена пользователем как образец алгоритма): простое совпадение количества по номенклатуре (характеристика игнорируется при `Справочник.Номенклатура.питПродажаПоСвободнойЦене`) + день + склад, БЕЗ пересчёта через рецептурные коэффициенты или модификаторы. Образцовый отчёт сравнивает чеки с оборотами `РегистрНакопления.ВыручкаИСебестоимостьПродаж` (а не напрямую с `питВыпускБлюд`) — конкретный запрос, реализующий то же сравнение против `Документ.питВыпускБлюд.Товары`, остаётся решением `design.md`, но принцип агрегации зафиксирован как факт.
- **Отрицательное отклонение** (в выпуске МЕНЬШЕ, чем ожидается по чекам) СЧИТАЕТСЯ проблемой: шаг ДОЛЖЕН показать явную строку с указанием номенклатуры, склада и величины дефицита; это отклонение блокирует прохождение шага (требуется действие «Переформировать выпуска» либо ручное исправление).
- **Положительное отклонение** (в выпуске БОЛЬШЕ, чем ожидается по чекам) НЕ СЧИТАЕТСЯ ошибкой — пользователь мог вручную добавить строки в документ выпуска сверх того, что получилось из чеков: шаг ДОЛЖЕН показать это как информационную пометку с подсказкой «возможно, добавлено вручную», не блокирующую прохождение шага.

#### Scenario: Отрицательное отклонение блокирует шаг
- GIVEN за 2026-07-05 по складу «Кухня-1» чеки предполагают выпуск 10 порций блюда «Плов», а в проведённых документах `питВыпускБлюд` фактически выпущено 7 порций
- WHEN мастер выполняет сверку чеков и выпусков на шаге «Формирование целевых документов»
- THEN шаг показывает строку отклонения с номенклатурой «Плов», складом «Кухня-1» и дефицитом 3 порции
- AND шаг «Формирование целевых документов» отмечается как не пройденный для «Кухня-1» на 2026-07-05

#### Scenario: Положительное отклонение не блокирует шаг
- GIVEN за 2026-07-05 по складу «Бар-1» чеки предполагают выпуск 5 порций блюда «Салат», а в проведённых документах `питВыпускБлюд` фактически выпущено 6 порций
- WHEN мастер выполняет сверку чеков и выпусков на шаге «Формирование целевых документов»
- THEN шаг показывает информационную пометку по номенклатуре «Салат» с подсказкой, что превышение, возможно, добавлено пользователем вручную
- AND это отклонение не блокирует прохождение шага «Формирование целевых документов» для «Бар-1» на 2026-07-05

### Requirement: Отсутствие рецептуры или производственного подразделения — не ошибка
Если для номенклатуры, проданной за выбранный день, нет рецептуры (`Документ.питРецептура`), либо выбранный склад не входит в состав производственных подразделений ни одной применимой рецептуры (ни напрямую, ни через вышестоящее подразделение), шаг «Формирование целевых документов» НЕ ДОЛЖЕН считать это ошибкой и НЕ ДОЛЖЕН подсвечивать это как проблему, требующую внимания пользователя.

#### Scenario: Склад не участвует в выпуске блюд
- GIVEN склад «Склад-Розница-1» не указан ни в одной рецептуре как производственное подразделение
- WHEN формирование целевых документов выполняется по чекам, относящимся к «Склад-Розница-1»
- THEN шаг не формирует по этому складу документ выпуска блюд и не показывает это как ошибку или предупреждение

### Requirement: Явный список ошибок формирования
Шаг «Формирование целевых документов» ДОЛЖЕН показывать пользователю явный список ошибок, возникших при формировании или проведении целевых документов за выбранный день по выбранным складам, классифицированный минимум по следующим категориям: нехватка марок алкоголя / отсутствие акта вскрытия тары; нехватка ингредиентов на остатке; отрицательные остатки, возникшие после выпуска; прочие ошибки непроведения.

#### Scenario: Нехватка акта вскрытия тары
- GIVEN блюдо, выпущенное 2026-07-05, использует алкоголь как ингредиент по рецептуре, и на складе не оприходовано достаточное количество марок этого алкоголя через `Документ.питАктВскрытияТарыАлкогольнойПродукции` (недостаточный остаток по измерениям «Организация+Склад+Номенклатура» в `РегистрНакопления.питОстаткиМарокИС`, ресурс `Количество`)
- WHEN обработка формирования целевых документов пытается сформировать/провести документ выпуска для этого блюда
- THEN шаг показывает ошибку категории «нехватка марок алкоголя / нет акта вскрытия тары» с указанием номенклатуры алкоголя и склада

#### Scenario: Нехватка ингредиента на остатке
- GIVEN блюдо использует по рецептуре ингредиент, которого недостаточно на остатке склада-производителя на 2026-07-05
- WHEN обработка формирования целевых документов пытается провести документ выпуска
- THEN шаг показывает ошибку категории «нехватка ингредиентов» с указанием номенклатуры ингредиента и склада

#### Scenario: Отрицательный остаток после выпуска
- GIVEN документ выпуска блюд проведён, но привёл к отрицательному остатку по одной из позиций на складе-производителе
- WHEN мастер собирает список ошибок шага «Формирование целевых документов»
- THEN шаг показывает предупреждение об отрицательном остатке по этой позиции для этого склада (в дополнение к отдельному шагу «Анализ отрицательных остатков»)

### Requirement: Отдельный список документов по открытым ценам
Шаг «Формирование целевых документов» ДОЛЖЕН показывать пользователю отдельный список документов `Документ.питВыпускБлюд`, сформированных по продажам с признаком «Открытые цены» (`Справочник.Номенклатура.питПродажаПоСвободнойЦене` на позициях чека), с указанием реквизита `ПЛ_Номенклатура` как маркера происхождения документа, и позволять открыть каждый такой документ для дозаполнения конкретной номенклатурой и проведения.

#### Scenario: Документ по открытой цене ожидает дозаполнения
- GIVEN за 2026-07-05 по складу «Бар-1» сформирован документ `питВыпускБлюд` с заполненным `ПЛ_Номенклатура`, товарные строки документа не содержат конкретной номенклатуры
- WHEN пользователь открывает список документов по открытым ценам на шаге «Формирование целевых документов»
- THEN документ отображается в списке как требующий дозаполнения (не проведён)
- AND пользователь может открыть документ прямо из мастера, заполнить конкретную номенклатуру и провести его

### Requirement: Критерий пройденности шага по открытым ценам — факт проведения
Документ `питВыпускБлюд`, сформированный по продаже с открытой ценой, СЧИТАЕТСЯ обработанным для целей этого шага исключительно по факту его проведения (`Проведён` = Истина). Заполненность реквизита `ПЛ_Номенклатура` НЕ ДОЛЖНА использоваться как критерий «требует внимания» — этот реквизит не очищается после дозаполнения документа и остаётся маркером происхождения на постоянной основе.

#### Scenario: Документ дозаполнен и проведён
- GIVEN документ `питВыпускБлюд` с `ПЛ_Номенклатура` заполнен, пользователь дозаполнил товарные строки конкретной номенклатурой и провёл документ
- WHEN мастер вычисляет, обработан ли этот документ
- THEN документ считается обработанным (несмотря на то, что `ПЛ_Номенклатура` остаётся заполненным)

### Requirement: Критерий пройденности шага «Формирование целевых документов»
Шаг «Формирование целевых документов» СЧИТАЕТСЯ пройденным для комбинации день+склад, когда: (а) чеки за этот день загружены полностью (см. `Requirement: Предусловие полноты загрузки чеков за день`); (б) в границах этого дня и склада не осталось записей `питУчетДанныхПродаж` с `ОбновленПриемник = Ложь` (все целевые документы актуальны); (в) все документы `питВыпускБлюд` за этот день и склад, включая документы по открытым ценам, проведены; (г) не осталось необработанных ошибок формирования (нехватка марок, нехватка ингредиентов, прочие ошибки непроведения) за этот день и склад; и (д) нет отрицательных отклонений сверки чеков и выпусков (см. `Requirement: Сверка чеков и выпусков`) за этот день и склад. Отрицательные остатки, возникшие после выпуска, НЕ блокируют этот шаг — они проверяются отдельным шагом «Анализ отрицательных остатков». Положительные отклонения сверки чеков и выпусков НЕ блокируют этот шаг.

#### Scenario: Остались непроведённые документы по открытым ценам
- GIVEN за 2026-07-05 по складу «Бар-1» есть один непроведённый документ выпуска по открытой цене
- WHEN мастер вычисляет статус шага «Формирование целевых документов»
- THEN шаг отмечается как не пройденный для «Бар-1» на 2026-07-05

#### Scenario: Всё проведено, но есть отрицательный остаток
- GIVEN все документы выпуска за 2026-07-05 по складу «Кухня-1» проведены без ошибок формирования, но выпуск привёл к отрицательному остатку одной позиции
- WHEN мастер вычисляет статус шага «Формирование целевых документов»
- THEN шаг отмечается как пройденный для «Кухня-1» на 2026-07-05
- AND отрицательный остаток отражается на отдельном шаге «Анализ отрицательных остатков»

### Requirement: Ограничение на удаление документа выпуска
Удаление документа `Документ.питВыпускБлюд` ДОЛЖНО быть доступно только пользователю с отдельной повышенной ролью (либо ролью «Полные права» — стандартный факт платформы 1С, не отдельное архитектурное решение этого change). Обычный пользователь мастера (роль «ответственный за склад/точку продаж», см. `../../proposal.md → ## Affected users`) этой возможности НЕ ДОЛЖЕН иметь. Конкретное имя и состав новой роли (объекты, право «Удаление» на конкретные объекты) в этом change НЕ фиксируются — решение `design.md`/`1c-metadata-manager`.

#### Scenario: Обычный пользователь не может удалить документ выпуска
- GIVEN пользователь с ролью «ответственный за склад/точку продаж» (без повышенной роли) открывает документ `питВыпускБлюд`
- WHEN пользователь пытается удалить документ (интерактивно или помечает на удаление)
- THEN операция недоступна из-за отсутствия права на удаление у текущей роли

#### Scenario: Пользователь с повышенной ролью может удалить документ выпуска
- GIVEN пользователь имеет отдельную повышенную роль (или роль «Полные права»)
- WHEN пользователь удаляет документ `питВыпускБлюд`
- THEN операция выполняется штатным механизмом платформы 1С без дополнительных ограничений этого change
